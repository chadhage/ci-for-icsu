# 5S

## TL;DR

5S — **Sort, Set in order, Shine, Standardize, Sustain** — is the visual workplace organization method born on Toyota's shop floor. Translated to engineering: clean the repos, organize the artifacts, automate the maintenance, codify the standard, audit the cadence. For CSAs, 5S applies to repos, subscriptions, dashboards, runbook libraries, CI catalogs, and ChatOps channels. Done well, 5S removes the silent waste of searching, mis-identifying, and re-creating — usually 5–15% of engineering hours in unmaintained estates. Treat it as recurring (quarterly) work, not a one-shot project.

## Why 5S

Disorganized estates leak time invisibly. Engineers can't find the canonical IaC module, so they fork; can't find the runbook, so they wing the incident; can't find the dashboard, so they build a redundant one. 5S compounds.

- Reduces search/identification waste (see [waste.md](./waste.md), categories: Motion, Defects, Over-processing).
- Eliminates orphaned resources (subs, repos, dashboards, ChatOps channels).
- Forces convention — which is its own [poka-yoke](./poka-yoke.md).
- Visually surfaces drift; engineers see the mess before customers do.

**Real example:** A customer's subscription estate had 1,400 resources of which 380 were orphaned (no owner, no tag, no traffic). A 5S sweep over 3 weeks tagged, retired, or transferred all 380. Savings: $94K/yr direct + 8 hours/week reclaimed from cost-allocation arguments. The Sustain step (monthly audit + automation) kept the count under 20 for 18 months running.

## What 5S is

| S                    | Japanese    | Engineering meaning                                                  |
| -------------------- | ----------- | -------------------------------------------------------------------- |
| 1. Sort              | Seiri       | Identify what's needed; remove what isn't.                          |
| 2. Set in order      | Seiton      | Put what's needed in a known, conventional place.                   |
| 3. Shine             | Seiso       | Continuously clean — automated lint, dependency upgrades, unit tests.|
| 4. Standardize       | Seiketsu    | Codify the convention — IaC modules, Policy, repo template.         |
| 5. Sustain           | Shitsuke    | Audit cadence + culture so it doesn't decay.                         |

The discipline is the *sequence*: skip Sort and you organize trash; skip Standardize and Sustain collapses.

## How to run 5S

1. **Pick the surface.** One repo, one sub, one runbook library. Not all at once.
2. **Sort.** What's used? What's orphan? Cut hard. Archive what might be needed; delete the rest.
3. **Set in order.** Define folder / tag / naming conventions. Move what's left.
4. **Shine.** Add automated upkeep — linters, dependency bots, scheduled cost audits, broken-link checks.
5. **Standardize.** Codify into a template (repo template, IaC module, Policy, dashboard pack).
6. **Sustain.** Quarterly audit; metric on dashboard; named owner; refresh cycle.
7. **Roll the template to peer surfaces.** Once one repo / sub is clean, propagate.

**Anti-pattern to avoid:** the "5S Saturday" — a one-time cleanup with no Standardize/Sustain. The mess returns within a quarter.

## When to use 5S

- Engagement intake — establish baseline order before anything else.
- After a merge / acquisition.
- When onboarding time exceeds a week.
- When orphaned resources / dashboards / channels become a topic.
- Quarterly maintenance cadence regardless.

## Where 5S shows up

- IaC monorepos.
- Subscription / resource-group estates.
- Runbook libraries.
- Dashboard / workbook catalogs.
- ChatOps channel inventories.
- AI agent prompt libraries.

## Who owns 5S

- **Platform / SRE leads** — primary surface owners.
- **CSAs** — facilitate first sweep; build templates; coach Sustain.
- **Each repo owner** — sustains their corner.
- **Manager / CoP lead** — funds the cadence.

## Examples

### Example 1 — Sub estate 5S

380 orphan resources retired; $94K/yr saved; sustained.

### Example 2 — IaC repo 5S

Repo of 142 modules → 38 canonical modules + 6 templates. PR cycle time fell 40%.

### Example 3 — Dashboard 5S

Workbook catalog: 240 dashboards → 31 canonical + an index. "Which dashboard?" questions fell to near zero.

### Example 4 — Runbook 5S

Wiki of 600 runbooks → 90 active, owner-tagged, refresh-dated. On-call resolution time fell 22%.

### Example 5 — ChatOps 5S

#-channels: 312 → 58. Search noise down; signal visibly up; channel naming convention enforced via bot.

### Example 6 — Prompt library 5S

Agent prompts in 14 forks consolidated to one versioned library with naming, owner, eval-suite link. Drift between forks eliminated.

### Example 7 — Dashboards Shine automation

Broken-link checker runs nightly; dead dashboards auto-flag at 30 days unused. Catalog stays curated without human attention.

### Example 8 — Standardize via repo template

New repos created from a hardened template; 5S compliance baseline at day 0.

### Example 9 — Sustain via quarterly metric

5S compliance % is a team KPI. Hovers ≥ 92%. Decline triggers a half-day refresh.

### Example 10 — 5S as engagement opening move

CSA standardized 5S sweep as week-1 of every engagement. Time-to-trust with customer engineers visibly accelerated — they noticed the cleanup before any other intervention.
